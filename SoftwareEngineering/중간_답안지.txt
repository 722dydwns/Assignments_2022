2. 소프트웨어 시스템의 초기 버전으로서 사용자 요구사항 도출, 사용자 인터페이스 시연, 설계 해결 방안 탐색 등에 활용되는 것은?  (1점)
  답) 프로토타입

3. 요구공학(requirements engineering) 초기 단계에 진행하여 프로젝트 진행여부를 결정하기 위한 것이며 시스템을 개발하는 것이 기술적으로 가능하고 조직에 도움이 되는지 평가하는 활동은?  (1점)
 답) 타당성 조사 

4. 요구사항의 출처, 요구사항 간의 관계, 요구사항과 설계 간의 관계를 명시적으로 관리하여 시스템 변경 시 변경의 영향을 파악할 수 있다는 것을 의미하는 특성(개념)은? (1점)
  답) 요구사항 추적성

5. 시스템의 사용자, 시스템 관리자, 시스템 소유자 등 시스템의 영향을 받는 사람 또는 조직을 통칭하는 용어는? (1점)
 답) 시스템 이해당사자 

6. 시스템 모델 중 시스템의 경계와 시스템과 연동되는 외부 시스템을 보여주는 모델로 시스템의 운영 환경을 기술하는데 사용되는 것은? (1점)
  답) 컨텍스트 모델 

7. 기능적 변환(transformation)으로 데이터를 처리하고 데이터가 변환을 통해 흘러가는 구조로 시스템을 나타내는 아키텍처는? (1점)
  답) 파이프 필터 아키텍처 

8. 공유 데이터베이스를 중심으로 도구들을 배치하여 도구 간 데이터를 공유하는 것이 용이한 아키텍처는? (1점)
  답) 저장소 아키텍처

9. 소프트웨어 프로세스 기본 활동 4가지를 수행되는 순서에 따라 나열하고 활동 내용을 간략하게 기술하시오. (6점)

 답)  소프트웨어 프로세스 기본 활동은 [명세 -> 개발(설계/구현) -> 검증 -> 진화] 4가지 활동으로 구성되어 있다. 

   '명세' 는 요구공학을 활용하여 요구사항을 분석하고 고객과 개발자가 소프트웨어의 기능, 운영 제약조건을 정의하는 단계이다. 
   '개발(설계/구현)' 은 소프트웨어를 설계하고 설계한 내용을 실체화하여 프로그램을 작성하는 단계이다. 
   '검증' 은 테스팅 단계로 소프트웨어가 고객의 요구사항에 맞는지 등을 확인하는 단계이다. 
   '진화' 는 유지보수 단계로서 고객, 시장의 변화하는 요구사항에 따라 소프트웨어를 수정 및 진화시키는 단계이다. 

10. V 모델이 무엇인지 간략하게 기술하시오. (3점)
  답) 소프트웨어 검증 활동이 폭포수 모델의 각 단계에 대응하는 것을 확인할 수 있는데, 
    테스트 계획이 테스팅과 개발 활동 사이를 어떻게 연결하는지 보여주는 V자형 구조로 구성된 모델이다.

11. 계획 주도(plan-driven) 개발과 애자일(agile) 개발 각각의 특징, 장단점, 적절한 적용분야를 비교하여 기술하시오. (8점)
  답) 계획주도 개발은 프로세스 단계별로 해당하는 산출물을 생성하도록 각 단계를 나누고, 각 단계의 산출물이 다음 단계에 활용되는 개발 기법이다. 
      이 개발 기법은 안정성 중심의 대규모 시스템 개발 분야에 적합하며 문서화에 중점을 두는 특징이 있다. 
            ex) 임베디드 시스템, 대규모 소프트웨어 시스템에 적용된다. 
       하지만, 계획주도 개발 기법이 소프트웨어의 신속한 개발에는 부적합하고 오버헤드가 크다는 단점이 존재하기 때문에 주목받게 된 것이 애자일 기법이다.
    애자일 기법은 설계 문서화를 최소화하여 개발자가 개발에 더 집중할 수 있도록 하는 데 중점을 두었으며,
    2-3주마다 새로운 증분을 인도하고 고객의 참여도를 높여 빠른 요구사항 변화에 대응할 수 있는 반복적인 개발 방식이다.
    이 기법은 중소규모의 제품 개발에 적합하고, 변화에 빠르게 대응할 수 있으므로 신속한 개발에 적합하다. 
            ex) 요구사항 쉽게 변하는 시스템, 중소 규모 시스템에 적용된다.  

12. XP의 사용자 스토리(user story)와 과업(task)이 각각 무엇인지 간략하게 기술하시오. (6점)
 답) 사용자 스토리는 시스템 사용자가 경험할 수 있는 일종의 사용 시나리오로서, 요구사항을 시나리오로 표현했다고 보면 된다.
  이 사용자 스토리를 기반으로 고객과 개발팀이 스토리 카드를 작성한 뒤, 그 내용을 다시  '과업(task)' 으로 세분화시켜 개발한다. 
    따라서 사용자 스토리가 전체를 포괄하는 내용물이라고 본다면, 과업은 더 세분화된 내용물이라고 보면 된다.

13. 스크럼(Scrum)에서 스프린트(sprint), 제품 백로그(backlog)가 각각 무엇인지 간략하게 기술하시오. (6점)
 답) 스크럼에서 스프린트는 개발에서 이루어지는 반복을 의미하며 보통 2~4주 주기로 구성된다.
    제품 백로그는 해야 할 일 목록을 정리한 To do list라고 보면 된다. 
   
14. 테스트 주도 개발(TDD: test driven development)이 무엇인지 설명하시오. (6점)
 답) 테스트 주도 개발은 코드 작성 전 테스트를 먼저 작성하는 것이다. 이때 테스트 자동화를 위해 자동화된 테스트 프레임워크 사용이 필수화된다. 

15. 기능적(functional) 요구사항, 비기능적(non-functional) 요구사항, 도메인(domain) 요구사항이 무엇인지 각각 설명하고 각 요구사항의 간단한 사례를 하나씩 제시하시오. (9점)
 답) 기능적 요구사항은 시스템이 제공해야 하는 기능. 서비스를 의미한다.
      이를테면 Mentcare 시스템에서 '사용자는 모든 클리닉에 대한 예약 리스트를 검색할 수 있어야 한다'는 것이 기능적 요구사항 사례이다.

     비기능적 요구사항은 기능 외적으로 시스템에 적용되는 '명세나 제약 조건'을 의미한다.
      이를테면 Mentcare 시스템에서 '시스템이 정규 업무 시간 동안 모든 클리닉에서 사용할 수 있어야 한다'는 제품 요구사항이 비기능적 요구사항 사례이다.
  
    도메인 요구사항은 응용 도메인에서 주어진 새로운 요구사항이나 기존 요구사항에 대한 제약 조건을 의미한다.  
     시스템 사용자의 특정 요구사항이 아니라 시스템의 응용 도메인에서 얻게 된다. 

16. 요구사항 도출 기법 중 문화기술적 연구(ethnography)가 무엇인지 간략하게 기술하시오. (4점)
   답) 관찰기법을 의미하는데, 운영(동작) 프로세스를 이해하고 이와 같은 프로세스를 지원하는 소프트웨어 요구사항을 얻기 위해 사용한다.
     현장에서 관찰을 통해 새롭게 요구사항을 도출할 수 있는 기법이다. 

17. 요구사항의 검증가능성(verifiability)이 무엇인지 간략하게 기술하시오. (2점)
  답)  시스템이 각 요구사항 만족시키는지 보여줄 테스트 작성 가능한지 점검하는 것을 의미한다. 

18. UML 유스케이스(use case) 모델에서 액터(actor)와 유스케이스(use case)가 각각 무엇인지 간략하게 기술하시오. (6점)
  답) 액터는 시스템과 '직접' 상호작용하는 모든 것을 의미하며 사람이나 시스템 자체가 될 수 있다.
     유스케이스는 시스템이 제공하는 기능을 의미하며 공통적인 사용자 목표와 관련된 시나리오 집합이라고 보면 된다. 

19. UML 액티비티(activity) 다이어그램에서 포크(fork), 조인(join)이 각각 무엇인지 간략하게 기술하시오. (6점)  // -6
  답)  데이터 처리와 관련된 행동 단위 흐름 표현 방식이다. 

20. UML 클래스 다이어그램에서 집합(aggregation)과 복합(composition) 관계가 각각 무엇인지 간략하게 기술하시오. (6점)
   답) 집합 관계는 한 객체가 다른 객체의 부분으로 구성되는 것을 의미한다.
        복합 관계는 집합 관계의 특수한 경우를 의미한다.

21. GoF의 Observer 디자인 패턴에서 ConcreteSubject가 무엇인지 간략하게 기술하시오. (4점)
    답) ConcreteSubject는 추상 객체를 상속받는 하위 객체를 의미하며,
      표시할 상태가 ConcreteSubject 객체에 유지되는데, 표시 추가, 삭제하는 오퍼레이션과 상태 변경 통지하는 오퍼레이션을 Subject로부터 상속받는다.  

22. 오픈소스 라이선스 중 GPL과 LGPL 각각의 특징을 간략하게 설명하시오. (4점)
  답) GPL 은 프로그램에서 GPL 소스를 일부라도 사용하는 것을 의미하는데, 변경 시 소스를 공개해야 하고 프로그램도 GPL 라이센스를 따라야 한다.
     LGPL은 완화된 GPL로서 LGPL 라이브러리를 사용할 경우 프로그램 소스를 공개할 필요는 없지만, 라이브러리르 수정한 경우에는 소스를 공개해야 하는 특징이 있다.  

23. 검증(validation) 테스팅과 결함(defect) 테스팅이 무엇인지 간략하게 설명하시오. (6점)
  답) 검증 테스팅은 소프트웨어가 고객 요구사항에 맞는지를 점검하는 테스팅이고,
       결함 테스팅은 소프트웨어 결합에 의해 제대로 동작하지 않는 부분을 발견하는 테스팅이다. 

24. 정적 V&V 기술과 동적 V&V 기술이 어떤 것이 있는지, 차이점은 무엇인지 설명하시오. (6점)
  답) 정적 V&V 기술로는 소프트웨어 인스펙션이 있다. 이 것은 프로그램을 실행시키지 않고 소스코드 등을 정적으로 분석한다.
    반면, 동적 V&V 기술로는 소프트웨어 테스팅이 있다. 이것은 프로그램을 실행시켜서 동작을 관찰하는 동적인 검증 기술이다.
    차이점은 프로그램 실행을 시키는지 여부에 있다. 

25. 동등 분할(equivalence partition)이 무엇인지 설명하시오. (4점)
 답) 동등 분할이란 분할 테스팅에서 입력 데이터 또는 출력 결과를 공통 특징 가진 그룹으로 그룹화(분할)하여 식별하는 것을 의미한다.
  동등 분할에서 분할된 각 분할에서 테스트 케이스를 선정하는 것을 분할 테스팅으로 보면 된다.
